# Internationalization (UI copy only)

A header **EN / ES** toggle localizes **UI copy only** — it never touches
user/business data (shop names, product descriptions, events stay as entered).

## Design: client-side, English SSR baseline
- English is the server-rendered/SSG baseline → town/store pages stay statically
  generated and English-indexed (SEO intact). Spanish swaps **client-side**.
- Locale persists in a `NEXT_LOCALE` cookie. No locale in the URL, no dynamic
  rendering forced on any route.

## Pieces
- `lib/i18n/locales.ts` — `LOCALES` (`en`, `es`), `Locale`, `isLocale`.
- `lib/i18n/en.ts` / `es.ts` — dot-path dictionaries (es reviewed by Claude).
  Missing es keys fall back to English automatically.
- `lib/i18n/dictionaries.ts` — `resolvePath()`, `getDictionary()`.
- `components/i18n/I18nProvider.tsx` — client context; SSR renders `en`, adopts
  the cookie on mount. Exposes `useT()`, `useLocale()`, `useI18n().setLocale`.
- `components/i18n/T.tsx` — `<T k="nav.towns" />`; usable inside server components
  (it's a client child), so pages stay server-rendered while text localizes.
- `components/i18n/LocalizedSearchInput.tsx` — for attributes (placeholder/aria)
  that need a string, not JSX.
- `components/layout/LanguageToggle.tsx` — the header EN/ES switch.

## How to translate another page/component
1. Add keys to `en.ts` (and the Spanish to `es.ts`).
2. In JSX text: `<T k="section.key" />`. In attributes: use a small client input
   (like `LocalizedSearchInput`) or `useT()` inside a client component.
3. Server components can render `<T>` directly; no need to make them client.

## Adding a language
Add the code to `LOCALES`, create `lib/i18n/<code>.ts`, register it in
`dictionaries.ts`. The toggle renders it automatically.

## Translation review with Claude
The Spanish set was authored/reviewed by Claude. To re-check after edits, hand
`en.ts` + `es.ts` to Claude and ask it to flag missing keys, mistranslations, or
tone issues (a `scripts/i18n-check.mjs` using the Anthropic API can automate this
later — `ANTHROPIC_API_KEY`).

## Coverage
Translated (EN/ES): the full **buyer journey** — header, footer, homepage, towns
+ TownFinder, town page, events, search + 3-layer empty exit + suggest form,
storefront, product page, cart, checkout, order success, `/shop`, and the
interactive bits (Add to cart, Share, RSVP, language toggle).

Not yet wrapped (render English gracefully; internal/less-critical): seller
dashboard, admin panel, auth (login/signup), buyer orders list/detail, and event
category labels. Migrate any of these by wrapping strings with `<T>` per the
pattern above.
